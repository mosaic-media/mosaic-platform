## Package tier model (correction to MEG-015 §02)

MEG-015 §02 — Repository Layout describes a two-tier model: `internal/platform/*` (private Platform code) and `internal/adapters/*` (with Postgres under `internal/adapters/postgres/`). Before implementation began on this repo, that layout was corrected to **three tiers**. Use this model, not MEG-015 §02's original layout:

1. **Core Platform** — `internal/platform/*`
   Domain, contracts, application services. Fully trusted, compiled in, defines the rules everything else follows.

2. **Built-in module** — `internal/modules/*`
   Infrastructure that implements Platform contracts using the same registration/manifest shape a future external Module would use (per MEG-006), but compiled in, required, and fully trusted — not sandboxed, not independently versioned, not optional.
   Postgres is the first example: it lives at `internal/modules/postgres/` (**not** `internal/adapters/postgres/`), registered through `internal/composition/builtin/` the same way an external Module would be discovered.

3. **External module** (future, per MEG-006)
   Product/domain capability packs — anime, manga, etc. Governed, independently versioned, discovered at runtime. **Not** part of this repo's initial scaffold.

`internal/adapters/` is reserved for things that are **not** module-shaped — helpers that don't implement a full contract surface on their own (filesystem utilities, crypto helpers). Do not put Postgres there.

**Current layout (verified against the tree, 2026-07-18):**

- `cmd/mosaic-platform/` — composition root (`main.go`: bootstrap → migrate → serve → graceful shutdown).
- `contracts/platform/v1/` — candidate public SDK contract surface. **Still only `doc.go`**; population is the first step of local slice 19.
- `internal/platform/` — Core Platform: `app`, `config`, `contracts`, `diagnostics`, `domain`, `events`, `policy`, `runtime`, `secrets`, `sessions`.
- `internal/modules/postgres/` — the built-in PostgreSQL module, with embedded `migrations/` (0001–0011).

- `test/contract/` — reusable, adapter-agnostic contract suite run against real PostgreSQL.
- `prompts/` — per-slice session prompts `13`–`20`, the plan of record for the remaining build sequence (the MAD-001 rework split plus the two final MEG-015 §12 slices).

## Non-negotiable rules (from MEG-015)

- **Dependency direction** (§02): dependencies point inward. Transport → Application services → Contracts/Domain. Adapters/Modules → Contracts → External systems. Domain must never import transport, adapter, or database packages. Application services may depend on Platform contracts, never on concrete Postgres (or other module) types.
- **Error categories** (§03): every contract error maps to one of `InvalidArgument`, `Unauthenticated`, `PermissionDenied`, `NotFound`, `Conflict`, `Unavailable`, `Internal`. Adapters/modules may keep driver-specific errors internally; application services and transports must only ever see these Platform categories.
- **Command handler order** (§04): every command handler follows the same sequence — validate command shape → authenticate caller → authorize via policy → open a `UnitOfWork` → load state through contracts → apply domain rules → persist state and outbox events in the same transaction → return a Platform result type.
- **GraphQL resolvers call services only** (§09): GraphQL is a transport and projection surface, not a persistence layer. Resolvers must call application command or query services — never open a database connection or query Postgres (or any module) directly.
- **Config reload classes** (§08): every configuration field declares a reload class — `Hot` (applies without restart), `Restart` (requires process restart), `Generation` (requires Supervisor to activate a new Generation), `Recovery` (applies only through recovery flow). Classify new config fields before adding them.

## Transaction contract shape correction (MAD-001)

`mosaic-architecture` main (commits `0ec5be8`, `cf90c8d`, `780d346`, docs session dated 2026-07-18) resolved the contradiction the Reference capability slice hit: the closed `Tx` interface (`Users()/Sessions()/Permissions()/Config()/Outbox()/Credentials()`) gave a new capability no way to join a transaction without editing Core Platform's `Tx` on its behalf, violating [MEG-006](https://github.com/mosaic-media/mosaic-architecture/blob/main/docs/engineering/guides/meg-006-module-platform/13-platform-guidelines.md) §13's "the Runtime should require no modification to support new capabilities." [MAD-001 — Transactional Store Extensibility](https://github.com/mosaic-media/mosaic-architecture/blob/main/docs/engineering/architecture/mad-001-transactional-store-extensibility/index.md) records the decision; MEG-015 §03 now carries it:

- `Tx` becomes a sealed, opaque transaction-scope marker with no accessor methods.
- Every store — Core Platform or capability — is resolved through a package-level generic function, `Store[T any](tx Tx) (T, error)`, not a named method (chosen over an extension registry, which loses type safety, and per-capability `UnitOfWork`, which can't guarantee atomicity — see MAD-001 §03 for the full comparison).
- Storage itself becomes a port: a `StorageAdapter` provides the `UnitOfWork` and binds each resolved store to the live transaction, so PostgreSQL becomes a swappable module rather than a privileged implementation.
- MEG-015 §12 also fixed an adjacent sequencing gap: promoting the proven contracts into `contracts/platform/v1` is now the *first step* of the Reference capability slice, not deferred to SDK readiness as originally written. (Those are MEG-015 §12's own slices 13 and 14 — in this repo's **local numbering** they sit at positions 19 and 20, because six MAD-001 rework slices, local 13–18, were interposed first. See Current status below.)

**The additive half of this resolution has now landed in code; the subtractive/migration half has not.** As of local slice 13 ("Core contracts rework" below), `internal/platform/contracts/` carries `Store[T any](tx Tx) (T, error)` (`store.go` — resolving through an unexported `resolveStore` that currently delegates to `Tx`'s accessors) and the `StorageAdapter` port (`storage_adapter.go`, `interface { UnitOfWork() UnitOfWork }` — **no implementor exists yet anywhere in the repo**). But `internal/platform/contracts/unit_of_work.go` still declares `Tx` **with all six accessors** (uniform resolution currently delegates to them and is valid only while they survive — sealing `Tx` is local slice 18), `contracts/platform/v1` is still empty but for `doc.go`, and every command handler that opens a transaction still calls `tx.Foo()` (no handler moved onto `Store[T]` yet). The full accessor-calling surface, re-verified by grep 2026-07-18: **seven** transactional command handlers (`internal/platform/app/create_local_user.go`, `authenticate_local_user.go`, `revoke_session.go`, `set_user_status.go`, `draft_config_version.go`, `validate_config_version.go`, `activate_config_version.go` — an earlier revision of this file said eight; seven is the recounted number), the postgres module's `outbox_atomicity_test.go`/`outbox_worker_test.go`, the reusable `test/contract/suite.go` harness, and the two six-accessor test fakes (`internal/platform/app/fakes_test.go`, `internal/transport/graphql/fakes_test.go`), plus `contracts/store.go`'s own accessor delegation by design. Per MAD-001's blast-radius assessment, the remaining migration is split into **five slices, one per originally-affected slice** (session prompts in `prompts/`, committed `bfdc648`):

- **Local slice 14 — PostgreSQL adapter rework** (`prompts/14-postgresql-adapter-rework.md`): make `internal/modules/postgres/unit_of_work.go`'s `tx` struct satisfy `Store[T]` against a live transaction (same store instances its six accessor methods already construct), provide the `StorageAdapter`-shaped entry point around `NewUnitOfWork(pool)`, migrate `outbox_atomicity_test.go`/`outbox_worker_test.go` onto `Store[T]`, extend `test/contract/suite.go` with a `Store[T]`-based mid-transaction-failure atomicity proof (keeping the accessor-based one), and prove `Store[T]`-write/accessor-read equivalence within one real transaction.
- **Local slice 15 — Application service skeleton rework** (`prompts/15-application-service-skeleton-rework.md`): migrate `create_local_user.go` (calls `tx.Users()`/`tx.Credentials()`/`tx.Outbox()`) onto `Store[T]`, plus the one-time `Store[T]` extension of the shared `internal/platform/app/fakes_test.go` fake that slices 16–17 rely on.

## Workflow

- Develop and commit directly on `main`. This repository does not use feature branches for Platform implementation work.
- Build one slice at a time, in the order defined by MEG-015 §12 — Build Sequence (as adjusted by the local numbering in Current status below). Do not start a slice whose prerequisites haven't landed.
- Every remaining slice (local 14–20) has a ready session prompt in `prompts/<nn>-*.md`. Start from the prompt; each one states its own prerequisites, scope, and exit criteria, and tells you to stop and report if a prerequisite slice isn't `[x]` here.
- Each slice must pass its MEG-015 §11 test gate before the next dependent slice begins.
- Run `go build ./...` and `go test ./...` before declaring any slice done.
- Commit per passing slice — one commit (or focused set of commits) per slice, not one commit for the whole build sequence.
- When ambiguity comes up, check Required Reading (and whatever spec the ambiguous area maps to) before guessing. Do not substitute assumption for a documented decision.

- [x] Core contracts rework (local slice 13 — uniform store resolution, MAD-001 expand step 1 of 6; session prompt `prompts/13-core-contracts-rework.md`) — landed the **additive** half of the MAD-001 transaction-shape correction in Core Platform code: `internal/platform/contracts/` now carries `func Store[T any](tx Tx) (T, error)` (`store.go`) and the `StorageAdapter` driven port (`storage_adapter.go`), the uniform, type-safe store-resolution surface MEG-015 §03's Contract Shape shows. **`Tx`'s six accessors (`Users()`/`Sessions()`/`Permissions()`/`Config()`/`Outbox()`/`Credentials()`) are deliberately left intact — sealing `Tx` into an opaque marker is a later rework slice's job, not this one.** Diff is confined to `internal/platform/contracts/` (two new files + `contracts_test.go`); nothing in `internal/modules/postgres/`, `internal/platform/app/`, or `internal/transport/graphql/` changed, and `contracts/platform/v1` is untouched (still `doc.go` only — its promotion is a later slice). **Exact shape chosen (later slices depend on reading this, not guessing):**
  - `StorageAdapter` is `interface { UnitOfWork() UnitOfWork }` — the driven port an engine (built-in PostgreSQL today, a future SQLite drop-in) implements to provide the transaction boundary and bind resolved stores to the live transaction. Only `UnitOfWork()` is exposed; binding is the adapter's internal responsibility, observable through `Store`, not a caller-invoked method. No adapter implements it yet (that is the PostgreSQL-adapter rework slice).
  - `Store[T](tx)` delegates to an **internal, unexported** resolver `resolveStore(tx Tx, typ any) (any, bool)`. The type parameter is carried into the resolver as a typed nil pointer `(*T)(nil)` — because a store contract `T` is an interface type, `var zero T` is a nil interface with no dynamic type, so a switch on `any(zero)` would match nothing; `(*T)(nil)` carries `*T` (`*UserStore`, `*EventOutbox`, …) as its dynamic type instead. `resolveStore` type-switches on that pointer type and, for each of the six known store contracts, **delegates to the matching `Tx` accessor** (`tx.Users()`, `tx.Outbox()`, …). This guarantees `Store[UserStore](tx)` returns the *identical instance* `tx.Users()` would for the same `tx` — the equivalence the test proves — and is a deliberate first-cut valid only while the accessors survive. An unknown `T` returns an `Internal`-category Platform `Error` (fail-loud), never a nil store; nothing at the caller-facing boundary is stringly-typed or requires a store to "ask permission" to join the transaction (MAD-001 §03 rejected option (a), the `any`-keyed extension registry, for losing exactly that — an internal, unexported switch keeps the property). **Migration note for the PostgreSQL-adapter / sealing slices:** `resolveStore` is the single place to repoint when `Tx` is sealed and its accessors removed — swap the accessor delegation for the `StorageAdapter`'s live-transaction binding; `Store`'s signature and every call site stay byte-for-byte unchanged. Command handlers moving from `tx.Foo()` to `Store[Foo](tx)` is a separate later slice (application-service rework) and is untouched here.
  - Test: `contracts_test.go` extends `mockTx` (not replaced) to hold one pointer-backed instance of each mock store as a field (`newMockTx()` constructs them), so both the accessors and `Store` hand back the identical instance — provable by pointer identity rather than empty-struct value equality. `TestStoreResolvesIdenticalInstanceAsNamedAccessor` proves `Store[T](tx) == tx.Foo()` for three store types (User/EventOutbox/Credential); `TestStoreFailsLoudlyForUnboundType` proves an unbound `T` (`SecretBroker`) yields an `Internal` error and a nil store.
  - Gate: `go build ./...`, `go vet ./...`, and `go test ./... -race` all green against real PostgreSQL 16 (embedded). Note: this machine had no Go toolchain or C compiler at slice start — Go 1.26.5 and a mingw-w64 gcc (for `-race`/cgo) were installed via winget this session to run the gate.
